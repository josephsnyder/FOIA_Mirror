Subject: FISCAL PENDING RPT;GIP ISSUE;CANCELLING RECONCILED ORDER

Category: 
  - Routine

Description:
============

 This patch will address the following 3 issues:
  
 1. MUMPS CODE ERROR IN PRCB2A - Fiscal indicate not all post items appear
    on the [PRCB RPT FISCAL PENDING ACTION] option when run.
  
 2. GIP (Generic Inventory Package) Issue - When zero QTY on Hand 
    inventory is increased the cost and inventory value remain zero.
  
 3. Issue with users being able to cancel a purchase card order that has
    been reconciled if not marked as final charge.
  
  
 ASSOCIATED NSR(s):
 ==================
 N/A
  
  
 Remedy:
 =======
 1. HD0000000136404 MUMPS CODE ERROR IN PRCB2A
         Duplicates: HD0000000270770 FGH (Florida, Georgia, HCS) - Fiscal
                     Pending Report incomplete.
  
                           
 2. HD0000000193675 GIP Issue - Zero QTY on Hand
  
  
 3. HD0000000274680 Issue with cancelling a PC Order that has been 
                    reconciled

 Remedy Overview:
 ================ 
 1. HD0000000136404 MUMPS CODE ERROR IN PRCB2A
  
 Problem:
 --------
 Fiscal indicates that not all general post items appear on [PRCB RPT
 FISCAL PENDING ACTION] option, only those flagged as Special Control Point
 - General Post Fund.
  
 Resolution:
 -----------
 Modify routine to search all 2237s and 1358s including those not flagged
 as SPECIAL CONTROL POINT.  Note in last column of report the type of
 SPECIAL CONTROL POINT if flagged, else NON (NONE).
  
 Routines: PRCB2A
  
  
 2. HD0000000193675 GIP Issue - Zero QTY on Hand
  
 Problem:
 --------
 User reports that when an item inventory has been reduced to zero (and
 therefore average cost goes to zero) subsequent uploading of scanned
 physical inventory will increase on hand, but average cost and inventory
 value remain at zero.
  
  
 Resolution:
 -----------
 Modify routines to set LAST COST in AVERAGE COST to calculate inventory
 value.
  
 Routines: PRCPUSA, PRCPUTRX, PRCPUUIP
  
  
 3. HD0000000274680 Issue with cancelling a PC Order that has been 
                    reconciled
  
 Problem:
 --------
 When user reconciles a Purchase Card order from [NEW PURCHASE ORDER]
 option with METHOD OF PROCESSING set as Purchase Card, they are able
 to cancel the order after the reconciliation processing if not marked as
 final charge.
  
  
 Resolution:
 -----------
 Modify routine to check for purchase order with METHOD OF PROCESSING
 set as Purchase Card.  The check already existed if order created under
 Purchase Card order options.
  
  
 Routines:  PRCHINQ
